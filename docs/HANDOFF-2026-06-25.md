# ResearchFinder Handoff - 2026-06-25

This document is intended for a new Codex conversation taking over the project.

## Project Summary

ResearchFinder is a private hosted research-discovery product for two users. Each morning it should:

1. Fetch new arXiv papers for each user's research profile.
2. Use the user's local Codex subscription through a Windows worker to generate ranked research ideas.
3. Show an inbox of papers and AI-generated ideas.
4. Let the user dispatch a promising idea into a deeper viability/prototype sprint.

The hosted app stores identity, profiles, source metadata, jobs, generated ideas, and UI state. The hosted app does **not** store Codex credentials or model API keys. Local Windows workers do AI work by calling the user's installed Codex CLI.

## Known Product Gaps

### 1. Daily Novelty Calibration

The first real daily inbox produced 10 ideas and all 10 had:

```text
noveltyStatus: needs_novelty_check
```

The first viability check also returned:

```text
verdict: needs_novelty_check
```

Root cause:

- The inbox prompt previously anchored Codex toward `needs_novelty_check`.
- The worker does not yet perform a real related-work search before assigning novelty status.
- The viability prompt mostly gives Codex the source paper, so it cannot honestly conclude novelty.
- `allowRelatedWorkSearch` exists on profiles but is not yet used by the v2 worker pipeline.

Design response:

- Spec: `docs/superpowers/specs/2026-06-25-daily-novelty-scan-design.md`
- Plan: `docs/superpowers/plans/2026-06-25-daily-novelty-scan.md`

Implement this next.

### 2. Dispatch Sprint Is Not a Real Prototype Sprint Yet

The dispatch UI offers durations like `1-3 hours` and autonomy like `fully auto`, but the current worker only asks Codex for a short structured viability JSON report. It does not yet create a local workspace, build a prototype, run experiments, or persist prototype artifacts.

This should be a separate design and implementation plan after daily novelty scan.

### 3. Hosted Daily Cron Still Needs Final Automation

Manual candidate cron command:

```powershell
curl.exe -sS -X POST -H "Authorization: Bearer <CRON_SECRET>" https://research-finder-lake.vercel.app/api/cron/candidates
```

Production still needs a reliable scheduler that can call `POST /api/cron/candidates` with the bearer secret. Vercel Cron may not support custom POST headers in the needed way; if not, use an external scheduler, GitHub Actions scheduled curl, or adjust the endpoint design.

## Mental Model for the Next Agent

ResearchFinder should feel like a private research operating system:

- Vercel app is the control plane.
- Neon Postgres is the shared durable state.
- Google auth gates the two users.
- The Windows Codex worker is the AI execution engine.
- The daily inbox should be evidence-backed, not merely plausible.
- The dispatch flow should eventually become a real build-and-evaluate sprint.

The immediate issue is not UI polish. The immediate issue is research reliability: novelty labels need actual search evidence before the product can guide paper-worthy project selection.
